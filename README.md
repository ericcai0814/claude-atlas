# claude-atlas

A read-only Tauri desktop dashboard that surfaces configuration drift across
the three tiers of Claude Code state on a developer machine: global
(`~/.claude/`), dotfiles source (`~/dotfiles/claude/`), and per-project
(`<repo>/.claude/`). Never mutates.

**Status**: v0.1.0 — Phase 1 shipped ([release notes](https://github.com/ericcai0814/claude-atlas/releases/tag/v0.1.0)).

---

## How this was built — spec-driven AI development

`claude-atlas` is a case study in **spec-driven AI development**. Every
requirement was authored as a formal `openspec` specification with WHEN/THEN
scenarios, edge cases, and explicit non-goals. AI agents implemented both the
Rust backend (`src-tauri/`) and the React/TypeScript frontend (`src/`) from
those specs. I authored the specs, designed the data model and command
contracts, made the architecture calls, reviewed and integrated the output,
and shipped the result.

I do not hand-write Rust or React in this codebase. The point of the repository
is to demonstrate that **a small, rigorous spec is what carries an
implementation across stacks** — not personal mastery of any one language.

## Capabilities (Phase 1)

1. **Multi-Tier Inventory Discovery** — enumerates skills, agents, hooks, and memory files across global, dotfiles, and per-project tiers; excludes runtime noise (`projects/`, `todos/`, `statsig/`, dotfile-prefixed entries)
2. **Four-State Drift Classification** — every artifact tagged `ok` / `drifted` / `broken` / `unmanaged` against the dotfiles source-of-truth
3. **Manifest-Driven Drift Detection** — parses each project's `<project>/.claude/dispatch.yaml` defensively (schema-agnostic), classifies declared artifacts as `satisfied` / `missing` / `excess` across three resolution tiers (project-local → global whitelist → dotfiles source)
4. **Unified Dashboard Presentation** — Overview drift summary + Symlinks / Projects / Plugins / Context drill-down tabs; severity-sorted (`broken` and `drifted` rows before `ok`); re-scans on tab activation, no persistent cache
5. **Non-Invasive Guidance** — drift surfaces copy-paste CTA shell commands (e.g. `cd ~/dotfiles && just restore`); the backend registers no command that writes to any Claude config path; permitted read-side OS affordances (`Reveal in Finder`, `Open in VS Code`) explicitly scoped in the spec

## Methodology — the openspec workflow

1. **Write the requirement as a spec** with WHEN/THEN scenarios and edge cases (e.g. "Symlink target missing" → classified `broken`; "Non-symlink where symlink is expected" → classified `drifted`; "Malformed manifest degrades gracefully" → record parse error, continue scanning)
2. **Define the data shape** in `ARCHITECTURE.md` — TypeScript interface + Rust struct signature in lockstep
3. **Hand the spec to an AI agent** for implementation
4. **Review, push back, and integrate** — when AI output deviated from the spec, the correction was to rewrite the spec section, not patch the code. This is the discipline that keeps the repo coherent across iterations
5. **Archive the change set** under `openspec/changes/archive/` once shipped, preserving the audit trail

The repository is as much an artifact of *how I work* as it is of *what got built*.
